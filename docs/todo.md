# Loggify — 仕様・要件の未決事項一覧

`docs/specs/draft-idea.md` および `docs/specs/tech-stack.md` を精査し、要件定義として不足・曖昧・矛盾している項目をすべて洗い出したもの。

## 1. エンティティ定義

### 1.1 User（エンドユーザー）

- [ ] 同一 Discord ユーザーが複数サーバーで認証した場合、ログはユーザー単位で横断検索できるか？ それともサーバー単位で独立か？
- [ ] ユーザーのブロック/BAN 機能の有無。特定 Discord ID をサーバー単位・組織単位・システム全体で認証拒否する仕組みは必要か？
- [ ] ユーザーのデータ削除リクエスト（GDPR 的な「忘れられる権利」）への対応方針。永久保存ポリシーとの整合性

### 1.2 SystemUser（システムユーザー）

- [ ] ER 図では `SystemUser └── 1:N ── Organization` だが、本文では「作成できる組織は1つまで」。「作成」は1つだが「所属」は N という意味か？ ER 図の関係は「所属」を表すなら、「作成」の制約はどこで表現するか明記が必要
- [ ] アカウント削除/退会フローが未定義。削除時に所有する組織・サーバーはどうなるか？
- [ ] アカウント復旧手段が未定義。Discord アカウントを失った場合の救済策（メール認証等）
- [ ] プロフィール情報として何を保持するか未定義（表示名、メールアドレス等）

### 1.3 Server（サーバー）

- [ ] 同一 Discord Guild ID が複数の組織から登録された場合の挙動。一意制約を設けるか、許可するか？
- [ ] サーバーの「無効化」と「削除」の区別はあるか？ 一時停止（認証フロー停止）の仕組みは必要か？
- [ ] サーバー削除後、対応する redirect_uri (`/api/callback/[server-uuid]`) へのアクセスはどう処理するか（404？カスタムエラー？）
- [ ] Bot がギルドから追放/退出した場合の検知・通知・自動無効化の仕組み
- [ ] サーバー設定のうち、登録後に変更可能なフィールドはどれか（Client Secret のローテーション等）

### 1.4 Organization（組織）

- [ ] 組織の削除フローが未定義。配下のサーバー・ログ・メンバーへの影響（カスケード削除 or ソフトデリート or 禁止）
- [ ] 組織のオーナー移譲フローが未定義。owner が退会・BAN された場合の扱い
- [ ] 組織名の一意性制約・命名ルール（文字数制限、使用可能文字等）
- [ ] 組織のプロフィール情報（アイコン、説明文等）の有無

### 1.5 Log（ログ）

- [ ] ログの DB スキーマが未定義。保存するフィールドの正確なリストとデータ型
- [ ] WebGL / Canvas フィンガープリントの保存形式（ハッシュのみか、生データも含むか）
- [ ] サーバー削除後の孤立ログの扱い。「紐付けは切れるがデータは保持」→ 誰がどうやってアクセスするのか？ システム管理者のみと記載があるが、元の組織管理者は閲覧不可になるのか？
- [ ] ログの更新・修正は発生するか（不変データとして扱うか）
- [ ] ログの重複判定。同一ユーザーが短時間に複数回認証した場合、全件記録するのか最新のみ上書きか

## 2. RBAC（ロールベースアクセス制御）

- [ ] `owner` / `admin` / `viewer` のパーミッションマトリクスが未定義
  - 各ロールが実行可能な操作の完全なリスト（サーバー追加/削除、ログ閲覧、メンバー管理、ASN ルール変更、組織設定変更等）
- [ ] `owner` は1組織に1人のみとあるが、owner 不在時（退会等）の昇格ルール
- [ ] ロールの変更（昇格/降格）は誰が実行できるか
- [ ] メンバーの除外は誰が実行できるか（owner のみ？ admin も？）
- [ ] 「システム管理者」（全サーバー・全組織のログを閲覧可能）の定義。Loggify 運営者のことか？ どのように認証・識別するか？

## 3. 認証・セッション管理

### 3.1 システムユーザー認証

- [ ] Discord OAuth2 と Passkey は「どちらか一方」でログイン可能か？ それとも Discord が必須でPasskey は補助的な位置づけか？
- [ ] Passkey の登録フローの詳細（ダッシュボード内の設定画面から登録？ 登録時に Discord 再認証が必要？）
- [ ] 同時ログインセッション数の制限はあるか
- [ ] セッション無効化（ログアウト）時の挙動。全デバイスログアウトの機能は必要か？

### 3.2 JWT 設計

- [ ] Access Token の有効期限（推奨: 15分〜1時間）
- [ ] Refresh Token の有効期限（推奨: 7日〜30日）
- [ ] Refresh Token のローテーション戦略（使用時に新規発行するか固定か）
- [ ] Token の revocation 機構（ブラックリスト方式？ DB チェック？）
- [ ] JWT の claim 構造（user_id, org_id, role 等、何を含めるか）
- [ ] Cookie の属性詳細（SameSite, Secure, Domain, Path）

### 3.3 エンドユーザー認証フロー

- [ ] GPS 許可が拒否された場合の挙動。現実装では `(0, 0)` にフォールバックするが、仕様として許容するのか？ 認証を拒否するのか？
- [ ] Turnstile 検証失敗時の挙動。リトライ可能か？ 回数制限は？
- [ ] 認証フローのセッション管理。iron-session を引き続き使うか、Go API 側に移すか？ マルチテナント化に伴う変更範囲
- [ ] 同一 Discord ID による同時認証（複数タブ等）の制御

## 凡例

- **未決（要議論）**: 関係者間で方針を決定する必要がある項目
- **未定義（要設計）**: 方針は暗黙的に決まっているが、詳細設計ドキュメントが不足している項目
- **矛盾（要修正）**: 仕様内で記載が矛盾している、または重複している項目
